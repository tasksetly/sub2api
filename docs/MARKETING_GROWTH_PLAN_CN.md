# Tasksetly 网站推广与增长执行计划

> 站点：https://ai.tasksetly.com
> 版本：v1.0
> 制定日期：2026-07-22
> 执行周期：90 天
> 配套技术规格：[MARKETING_TECH_SPEC_CN.md](MARKETING_TECH_SPEC_CN.md)

## 1. 结论先行

Tasksetly 当前不适合马上购买大规模流量。第一优先级不是“多发广告”，而是先补齐公开价格与模型信息、接入文档、可信度、搜索引擎基础、转化归因和新用户首次调用路径。否则访问者无法在注册前判断价格、可用模型、接入难度和服务保障，运营方也无法知道流量在哪一步流失。

建议采用以下顺序：

1. 第 1-2 周完成转化与数据基础，达到“可推广”门槛。
2. 第 3-4 周邀请 30-50 名目标开发者做受控冷启动，修复接入阻力。
3. 第 5-8 周通过高意图教程、开发者社区、开源生态合作和邀请返利获得稳定自然流量。
4. 第 9-12 周只对已经验证过转化的关键词、内容和合作方追加预算。

首阶段建议只打一个明确定位：

> Tasksetly 是面向 AI 编程用户和小型研发团队的一站式 API 接入服务，一个 API Key 接入 Codex、Claude Code、Gemini CLI 等常用工具，统一计费、用量和密钥管理。

“稳定”“低价”“官方源”“免费试用”等承诺必须有实时数据、明确规则或授权依据后再公开使用，不能只写口号。

| 项目 | 当前事实 | 推广影响 |

| 价格与状态 | 可用渠道和渠道监控功能存在，但只对登录用户开放 | 最强的购买证据被挡在登录墙后 |

| 防滥用 | Turnstile、邀请限制、优惠码、风控中心均未启用 | 一旦提供可用试用额度，批量注册风险较高 |
| 数据归因 | 源码及线上返回 HTML 未发现完整的流量到激活/付费事件链路 | 无法计算渠道 CAC，也无法判断页面或渠道是否有效 |
| 产品内接入 | 已有 API Key 管理、客户端配置片段、Codex/Claude Code/Gemini 等工具选项 | 激活基础较好，应把这套能力前置成清晰的新手任务流 |

## 3. 推广前必须确认的业务输入

以下数据不影响本计划开始执行，但在投入广告费或公开返佣前必须由负责人确认：

| 输入 | 负责人需要给出的答案 | 用途 |
| --- | --- | --- |
| 上游授权 | 各上游是否允许转售、共享、转换为 API 或面向第三方提供服务 | 决定哪些能力可以公开宣传，降低封禁与合同风险 |
| 主推地区 | 中国大陆、港澳台、东南亚或全球；是否有明确禁止地区 | 决定语言、支付、支持时间和投放平台 |
| 单位经济 | 各模型/分组毛利率、平均首次充值、30/90 天复购、退款率 | 决定 CAC 上限、试用额度和返利比例 |
| 主推产品 | 余额按量付费、订阅套餐或两者并行 | 决定首页价格结构和转化事件 |
| 可验证优势 | 近 30 天成功率、延迟、可用模型、价格、客服响应 | 决定首页证据和内容选题 |
| 免费试用预算 | 每个有效新用户允许消耗的真实上游成本 | 决定额度、模型范围和反作弊强度 |
| 客服承诺 | 服务渠道、工作时间、首响时间、退款处理规则 | 决定是否具备放量能力 |
| 月度预算 | 技术、内容、合作方、试用成本和广告费分别可投入多少 | 决定第 5 周后的扩张速度 |

在数据未知时，本计划中的百分比是冷启动阶段的目标区间，不是收入预测。

## 4. 目标用户与定位

### 4.1 第一优先用户：高频 AI 编程开发者

- 已使用 Codex CLI、Claude Code、Gemini CLI、OpenCode 等工具。
- 痛点是多个供应商账号、支付和 API Key 分散，接入配置繁琐，成本与用量不可见。
- 决策周期短，愿意通过一段可复制配置和少量试用快速验证。
- 最看重真实可用模型、最终价格、成功率、延迟、退款与客服，而不是抽象品牌故事。

### 4.2 第二优先用户：2-20 人研发团队

- 需要统一采购、按成员或项目分配密钥和额度、查看明细。
- 决策周期较长，但客单价和留存潜力更高。
- 需要账单、发票、权限、配额、服务支持和隐私说明。

### 4.3 后置用户：集成商、培训社群和内容创作者

- 可以成为分销或联盟合作伙伴，但容易带来低质量注册和套利。
- 只有在激活漏斗、返利风控和客服容量验证后再开放。

### 4.4 暂不优先

- 只寻找免费 ChatGPT 网页版的泛用户。
- 与 API、编程工具无关的大范围娱乐流量。
- 以“最低价”为唯一诉求、没有持续使用意图的羊毛流量。

## 5. 信息架构与核心话术

### 5.1 推荐首页首屏

品牌名：

> Tasksetly AI API

主标题：

> 一个 API Key，接入 Codex、Claude Code 与 Gemini CLI

副标题：

> 按量计费，统一查看模型价格、调用用量和密钥配额。复制配置即可开始调用，无需分别管理多个服务商账号。

主 CTA：

> 查看价格与模型

次 CTA：

> 开始接入

仅在试用额度、规则和风控均已配置时，才把次 CTA 改为“免费试用”。“立即开始”不应只跳到登录页，应根据来源进入对应工具的快速接入路径。

### 5.2 首页需要按顺序回答的问题

1. 这是给谁用的，支持哪些工具和协议？
2. 当前有哪些模型，最终计费价格或倍率是多少？
3. 如何接入，最短需要几步？
4. 最近 24 小时/7 天的成功率、延迟和渠道状态如何？
5. 数据、日志、退款、支持和服务边界是什么？
6. 免费试用或首充优惠的具体规则是什么？

### 5.3 可用证据

- 公开且更新时间明确的模型与价格表。
- 状态页的 24 小时和 7 天成功率、P50/P95 延迟、最近事件。
- 60 秒从注册到首次 `200 OK` 的无剪辑录屏。
- Codex、Claude Code、Gemini CLI 分别可复制的配置。
- 用量明细、密钥配额和团队管理的真实产品截图。
- 客服渠道、工作时间、退款与故障处理规则。
- 经允许发布的用户案例，明确场景、调用规模和结果。

不得发布无法验证的“100% 官方”“零封号”“永不宕机”“全网最低价”等绝对表述。

## 6. 北极星指标与转化漏斗

### 6.1 北极星指标

核心指标使用“每周新增激活开发者”，定义为：

> 新注册用户在 24 小时内完成邮箱验证、创建 API Key，并产生至少 3 次非后台测试、真实客户端发起的成功 API 调用。

商业结果同时观察：

- 7 天内首付费用户数。
- 每周付费活跃开发者数。
- 30 天贡献毛利，而不是只看充值流水。

### 6.2 漏斗定义与冷启动目标

| 阶段 | 精确定义 | 第 1-4 周目标 | 第 5-12 周目标 |
| --- | --- | ---: | ---: |
| 有效访问 | 非机器人、停留超过 5 秒或发生一次交互的落地页会话 | 建立基线 | 渠道分层增长 |
| 首页 CTA | 点击价格、文档或开始接入 | 访问的 10%-15% | 12%-20% |
| 注册完成 | 后端成功创建用户 | 访问的 5%-8% | 6%-12% |
| 邮箱验证 | 注册后 24 小时内完成验证 | 注册的 70%-80% | 80% 以上 |
| 创建 Key | 验证后 24 小时内创建首个 Key | 已验证的 45%-60% | 60% 以上 |
| 首次成功调用 | 创建 Key 后 24 小时内出现首个真实成功调用 | 创建 Key 的 50%-65% | 65% 以上 |
| 激活 | 24 小时内达到 3 次真实成功调用 | 注册的 20%-30% | 30%-40% |
| 首次付费 | 激活后 7 天内完成有效支付 | 激活的 12%-20% | 18%-30% |
| D7 留存 | 激活后第 7 天仍有真实成功调用 | 激活的 25% 以上 | 35% 以上 |

这些目标必须按来源、内容、设备、地区和工具类型拆分，不能只看全站平均数。

### 6.3 质量护栏

- 有效请求成功率、延迟和上游错误必须按模型单独观察，不能用全站平均掩盖问题。
- 未取得至少连续 7 天真实数据前，不对外承诺具体 SLA。
- 退款率、支付拒付率、滥用账户占比和客服工单量必须与增长数据一起评审。
- 活跃增长不得以大幅亏损试用或不可持续返佣换取。

## 7. 90 天执行路线图

### 阶段 0：第 1-2 周，达到可推广门槛

目标：让陌生开发者不需要联系客服，也能判断产品是否适合自己，并能被完整归因到首次调用和付费。

产品与配置：

- 明确按量余额、订阅套餐和兑换码的实际售卖关系。
- 配置客服入口、文档地址、服务条款、隐私政策、退款政策、支持地区和服务特定条款。
- 启用忘记密码；核对邮件送达率和垃圾箱情况。
- 若提供试用，先启用 Turnstile、邮箱验证、基础注册频控和试用风控。
- 校验首页“免费试用”与后台默认余额/订阅完全一致。
- 暂不全量开启返利，只对内部测试账号和首批合作方灰度。

开发交付：

- 公共价格、模型、状态和快速接入页。
- 正确的 `robots.txt`、`sitemap.xml`、Canonical、页面描述、分享卡片和 404。
- UTM/来源保存、核心前端事件和后端激活/支付事件。
- 新用户任务清单：验证邮箱、选择工具、创建 Key、复制配置、发起测试、完成真实调用。

内容交付：

- Codex CLI、Claude Code、Gemini CLI 三份 3 分钟接入文档。
- 价格与计费说明、常见错误、退款与支持说明。
- 一段 60-90 秒真实接入视频。

阶段退出条件：第 12 节“推广上线门槛”全部通过。

### 阶段 1：第 3-4 周，受控种子用户

目标：用真实目标用户验证定位、接入和付费，不追求访问量。

- 定向邀请 30-50 名正在使用 AI 编程工具的开发者。
- 来源优先为现有用户、GitHub 贡献者、熟悉的开发者社群和 1 对 1 邀请。
- 至少完成 10 次 20 分钟用户访谈或屏幕共享接入观察。
- 每次只记录用户卡住的具体步骤，不用“感觉不错”代替行为数据。
- 每 2-3 天发布一次修复，优先处理无法理解价格、无法创建 Key、配置错误和首调失败。

阶段目标：

- 至少 20 个有效注册。
- 至少 8 个激活开发者。
- 至少 3 个首付费用户，或明确记录不付费的前三个原因。
- 80% 的种子用户无需人工改配置即可完成首次调用。

### 阶段 2：第 5-8 周，建立自然获客

目标：形成“搜索/社区内容 -> 对应工具落地页 -> 首次调用”的稳定链路。

- 每周发布 2 篇高意图教程、1 个短视频、1 份状态或产品更新。
- 在 GitHub Discussions、Linux.do、V2EX、掘金、SegmentFault、B 站等渠道选择符合规则的平台发布解决问题型内容。
- 每篇内容使用独立 UTM，不在多个平台复制完全相同的硬广。
- 联合 3-5 个开源客户端、教程作者或小型技术社群制作真实接入内容。
- 灰度邀请 5-10 名合作方使用返利链接，冻结期内人工检查异常注册和支付。
- 建立品牌词、工具词和错误排查词的搜索内容集群。

阶段目标：

- 自然与社区流量占新增有效访问的 60% 以上。
- 每周新增激活开发者连续 4 周增长。
- 至少 3 个内容页面能持续带来注册或首次调用。
- 返利渠道的激活率不低于全站平均的 70%。

### 阶段 3：第 9-12 周，验证付费扩张

目标：只给已经证明能产生激活和毛利的渠道增加预算。

- 对“Codex API 接入”“Claude Code API 配置”“Gemini CLI API”等高意图精确词做小额搜索测试。
- 每个关键词组使用对应落地页，禁止所有广告统一进入首页。
- 对已经产生至少 3 个付费用户的内容创作者追加合作，不按浏览量单独结算。
- 对 D7 留存和毛利达标的邀请来源提高额度；低质量来源降权或停用。
- 建立团队版销售入口，针对 2-20 人研发团队做 5-10 个定向试点。

阶段目标不使用固定流水数字，而由单位经济决定：

> 单渠道获客成本不高于该渠道付费用户 90 天贡献毛利的 35%，且连续两个周 cohort 达标后才允许每周增加不超过 20% 的预算。

## 8. 渠道优先级与具体打法

| 优先级 | 渠道 | 为什么 | 主要内容 | 核心指标 |
| ---: | --- | --- | --- | --- |
| 1 | 高意图 SEO | 用户已经在找具体 API 或配置方法 | 工具接入、错误排查、价格、模型、状态 | 自然激活数、页面到激活率 |
| 2 | 开发者社区 | 可快速获得高质量反馈和口碑 | 完整教程、真实测试、问题复盘 | 有效讨论、注册、激活 |
| 3 | GitHub/开源生态 | 与产品受众高度重合，信任成本低 | 示例配置、集成 PR、模板仓库 | 引荐激活、集成使用量 |
| 4 | 老用户邀请 | 来源可信，边际成本可控 | 双方权益、团队邀请 | 被邀请用户毛利、作弊率 |
| 5 | 技术创作者/社群 | 可规模化解释复杂接入 | 视频、直播、联合教程 | 每位合作方 CAC、D7 留存 |
| 6 | 精确搜索广告 | 可快速验证高意图关键词 | 一词一页、价格与快速接入 | 付费 CAC、90 天毛利 |
| 暂缓 | 信息流、泛科技媒体、批量群发 | 意图弱，难归因，易损伤品牌 | 不在前 90 天重点投入 | - |

### 8.1 SEO 内容集群

必须先建设的公共页面：

- `/pricing`：最终价格、倍率、计费例子、更新时间和例外条件。
- `/models`：模型、协议、上下文、工具兼容性和当前状态。
- `/status`：公开成功率、延迟和事件，不暴露上游账号信息。
- `/docs/quickstart`：从注册到首次调用。
- `/guides/codex-cli`、`/guides/claude-code`、`/guides/gemini-cli`。
- `/faq/billing`、`/faq/errors`、`/legal/*`。

首批高意图选题：

1. Codex CLI 自定义 API Base URL 的完整配置与验证。
2. Claude Code 使用第三方 API 时常见的 401、429 和模型名错误。
3. Gemini CLI API Key 与模型配置示例。
4. 一个 Key 同时管理多个 AI 编程工具的方法。
5. 按量计费与官方月度订阅的成本计算方式。
6. 如何读取 token、缓存和请求费用明细。
7. API 流式输出中断的排查清单。
8. 团队如何按项目设置 API Key 配额。
9. 模型切换时如何保持配置和账单一致。
10. 真实状态页应该怎么看成功率、延迟和故障。

每篇文章只设置一个主 CTA，例如“复制 Codex 配置”或“查看当前模型价格”，避免同时要求关注、加群、注册、充值。

### 8.2 社区发布原则

- 先解决一个具体问题，再自然给出 Tasksetly 示例。
- 明确作者与服务方的关系，不伪装成第三方用户推荐。
- 发布真实价格、时间、模型和测试环境，历史数据注明采样区间。
- 尊重各平台广告、返利链接和邀请码规则。
- 不购买虚假评论、不批量私信、不刷榜、不使用无法验证的对比结论。
- 评论区技术问题在一个工作日内回复，故障问题直接链接状态页。

### 8.3 开源与合作伙伴

- 为已实际支持的客户端提供最小配置示例和一键复制，不做虚假“官方合作”表述。
- 优先与能带来真实使用行为的教程作者合作，而不是只看粉丝数。
- 合作结算优先采用“小额基础制作费 + 激活/首付费奖励”，避免纯注册 CPA。
- 每位合作方使用独立 `utm_campaign`、`utm_content` 和返利码。
- 合作内容必须保留至少 90 天，价格变化时可更新。

### 8.4 邀请返利建议

现有系统可以直接支持首轮灰度，推荐初始规则：

- 返利以被邀请用户的实际到账净收入或可计算毛利为基础，不对免费额度、退款订单、管理员赠送和异常支付返利。
- 返利冻结 7-14 天，覆盖退款和拒付窗口。
- 有效返利期先设 90 天，不建议冷启动即承诺永久返利。
- 设置单个被邀请人的返利上限。
- 返利总成本不超过被邀请用户 90 天预计贡献毛利的 25%-30%。
- 首月仅邀请审核过的合作方，确认作弊率和留存后再向所有用户开放。

## 10. 试用、优惠与预算

### 10.1 试用设计

- 试用的目标是让用户完成一次真实工作流，而不是单纯领取余额。
- 额度应覆盖约 15-30 分钟真实体验，并按主推模型的实际成本倒推。
- 明确有效期、适用模型、并发、是否可叠加优惠及退款规则。
- 邮箱验证、Turnstile、IP/设备频控和异常支付检测上线后再开放。
- 风控只保存业务所需信息，隐私政策说明目的、期限和申诉方式。
- 不允许把试用额度转出或产生邀请返利。

### 10.2 三档月度推广预算

开发成本和上游调用成本应单列，不能藏在广告预算里。

| 档位 | 月度现金预算 | 适合阶段 | 建议分配 |
| --- | ---: | --- | --- |
| 冷启动 | ¥3,000-¥8,000 | 第 1-4 周 | 内容/视频 40%，种子用户试用 25%，小型合作 25%，工具 10% |
| 验证增长 | ¥15,000-¥30,000 | 第 5-8 周 | 内容 25%，创作者与联盟 35%，试用 15%，精确搜索测试 15%，工具与素材 10% |
| 扩张 | ¥50,000 以上 | 第 9 周后 | 只按已验证的渠道 CAC 与毛利动态分配，不预设平台比例 |

### 10.3 CAC 和停损规则

先计算：

```text
90 天贡献毛利 = 90 天净收入 - 上游调用成本 - 支付手续费 - 退款/拒付 - 可变客服成本
付费 CAC 上限 = 90 天贡献毛利 x 35%
激活 CAC 上限 = 付费 CAC 上限 x 激活到付费率
```

渠道满足以下任一条件即暂停并复盘：

- 花费达到 2 个付费 CAC 上限仍没有付费用户。
- 累计 50 个注册后，激活率仍低于全站平均的 70%。
- 退款、拒付、滥用或客服负担显著高于全站平均。
- 合作方无法提供真实来源或出现误导宣传。

连续两个周 cohort 的 CAC、D7 留存和毛利均达标后，预算每周最多增加 20%。

## 11. 实验清单

低流量阶段不做复杂 A/B 测试。每个来源 cohort 只改变一个主要变量，先用行为和访谈找大问题。

| 优先级 | 假设 | 变更 | 主指标 | 成功条件 |
| ---: | --- | --- | --- | --- |
| 1 | 公开价格可提高有效注册 | 首页先展示价格和模型，再注册 | 访问到激活率 | 不降低流量质量且提升 20% 以上 |
| 2 | 工具专属入口比通用首页有效 | Codex/Claude/Gemini 各自落地页 | 落地页到首次调用 | 比通用首页高 20% 以上 |
| 3 | “查看价格”比“立即开始”更符合决策路径 | 更换主 CTA | CTA 到注册/激活 | 激活数提升且跳出下降 |
| 4 | 接入任务清单可减少首调流失 | 注册后显示 5 步任务 | 注册到激活率 | 提升 15% 以上 |
| 5 | 一键测试请求能快速建立信心 | 创建 Key 后提供安全测试 | 首调时间、激活率 | 中位首调时间下降 30% |
| 6 | 透明状态提升付费信任 | 付款前展示状态和事故记录 | 激活到首付费 | 提升且退款不增加 |
| 7 | 双方权益比单边返利更有效 | 邀请人返利 + 新用户小额权益 | 邀请激活率、毛利 | 毛利为正且作弊率可控 |

所有实验需记录开始时间、适用来源、样本、版本、结果和决策。没有达到最低样本时标记为“信息不足”，不能强行宣布胜负。

## 12. 推广上线门槛

以下项目全部通过后，才进入公开社区推广或付费投放：

- [ ] 首页明确主推工具、价格入口、模型入口和接入入口。
- [ ] 免费试用文案与真实额度、有效期和限制一致；不提供试用则删除该承诺。
- [ ] 文档可在未登录状态访问，三类主推工具均能照文档完成调用。
- [ ] 公开模型和价格包含更新时间、计费单位和例子。
- [ ] 公开状态页不泄露上游账号或内部架构。
- [ ] `robots.txt`、`sitemap.xml`、Canonical、OG 和真实 404 正确。
- [ ] 客服、服务条款、隐私、退款、支持地区和服务边界有有效正文。
- [ ] 忘记密码和邮件验证完整走通，邮件送达率已抽查。
- [ ] 试用开启时，Turnstile、注册频控和异常账户处置流程已上线。
- [ ] UTM 能从首次访问关联到注册、首次成功调用和支付。
- [ ] 支付成功事件以后端 webhook/订单为准，重复回调不会重复计数。
- [ ] 已用至少 5 个不同设备/网络的测试账号跑通完整漏斗。
- [ ] 客服有明确负责人、首响目标和故障公告流程。
- [ ] 上游授权、商标使用和公开宣传表述已经过负责人确认。
