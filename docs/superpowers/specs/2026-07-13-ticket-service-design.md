# 工单服务设计

## 目标与范围

为已登录用户提供内置、可追踪的支持工单闭环，并让管理员能够在后台统一处理。首期包括用户创建与查看工单、双方纯文本留言、管理员筛选与状态处理。

首期不包括邮件或站内通知、实时更新、附件上传、消息编辑/删除、管理员个人指派、客服角色、可配置分类或外部客服系统集成。

## 已确认的业务规则

- 工单状态：`pending`（待处理）、`in_progress`（处理中）、`resolved`（已解决）、`closed`（已关闭）。
- 管理员可将任何未关闭工单更新为 `in_progress`、`resolved` 或 `closed`。
- 用户可在 `pending`、`in_progress`、`resolved` 工单中追加回复；对 `resolved` 工单回复时，状态自动切换为 `in_progress`。
- `closed` 工单不可追加消息或再打开。
- 创建工单时用户必须提供主题、分类和首条正文。内置分类为：`account`、`billing`、`api`、`usage`、`other`。
- 优先级仅由管理员设置：`low`、`normal`、`high`、`urgent`；新工单默认 `normal`。
- 所有管理员共享队列，不做个人指派；每条消息记录真实发送者。
- 消息均为纯文本。数据模型不排斥后续增加附件实体或外部对象存储关联。

## 架构与持久化

新增独立的工单领域模块，遵循现有 Ent、仓储、服务、处理器和路由分层。

### Ticket

- `id`
- `user_id`：提交者
- `subject`
- `category`
- `priority`
- `status`
- `last_activity_at`
- `created_at`、`updated_at`

`Ticket` 与一个用户关联，并与多条 `TicketMessage` 关联。列表按 `last_activity_at` 倒序分页，管理员可按状态、分类、优先级及用户关键词筛选。

### TicketMessage

- `id`
- `ticket_id`
- `sender_user_id`
- `sender_role`：`user` 或 `admin`
- `content`
- `created_at`

消息不可编辑或删除，形成完整对话与审计记录。

### 服务层职责

`TicketService` 是状态机和权限规则的唯一入口：创建时写入首条消息，回复时验证访问权与关闭状态，状态更新时验证管理员权限和合法枚举。每次创建消息或变更状态均更新工单的 `last_activity_at`。

## HTTP API

所有接口使用已有登录鉴权与统一响应/错误格式。

### 用户接口

- `GET /api/v1/tickets`：分页列出当前用户的工单。
- `POST /api/v1/tickets`：创建工单及首条消息。
- `GET /api/v1/tickets/:id`：读取当前用户的工单详情和消息。
- `POST /api/v1/tickets/:id/messages`：向当前用户的未关闭工单追加消息。

### 管理员接口

- `GET /api/v1/admin/tickets`：分页、筛选和搜索所有工单。
- `GET /api/v1/admin/tickets/:id`：读取工单、用户概要和完整消息。
- `POST /api/v1/admin/tickets/:id/messages`：管理员回复。
- `PATCH /api/v1/admin/tickets/:id`：更新状态和/或优先级。

服务层在用户访问非本人资源时返回授权错误；对关闭工单追加消息、无效分类/状态/优先级或超出文本长度的请求返回可展示的业务错误。

## 前端体验

- 用户端增加“我的工单”入口、工单列表、创建表单及带对话时间线的详情视图。
- 管理员端增加“工单管理”入口、筛选表格和处理详情视图；管理员可回复、更新优先级和状态。
- 页面遵循既有 Vue 路由守卫、API 模块、组件样式和中英文 i18n 组织方式。
- 关闭工单时隐藏或禁用回复输入框，并清楚展示状态。
- 首期依靠进入页面后的加载获取最新数据，不展示未读数，不做轮询或推送。

## 约束与验证

- 主题、正文和筛选参数受服务端长度与枚举校验，防止无界存储与无效查询。
- 前端复用服务端错误信息并提供本地必填校验。
- 后端测试覆盖状态机、用户资源隔离、关闭工单不可回复、管理员筛选和写入校验。
- 前端测试覆盖创建、回复、关闭后的禁用状态、管理员筛选与更新动作。

## 后续扩展

附件可通过新建 `TicketAttachment` 并连接对象存储实现；通知可订阅“新消息”和“状态变化”领域事件；个人指派则可在 `Ticket` 上增加 `assignee_id`，而不改变消息或现有状态机接口。
